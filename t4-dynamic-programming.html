Dynamic Programming

Given a known model, DP computes optimal values and policies by repeatedly applying the Bellman equations as update rules. The two algorithms — policy iteration and value iteration — are exam staples, and the synchronous vs. in-place distinction decides your arithmetic.

1. The core idea

The Bellman equations are fixed-point equations. Turn the “=” into a “←” and iterate, and the values converge to the true solution. DP is model-based (needs \(p(s',r\mid s,a)\)) and bootstraps (updates estimates from other estimates).

💡
Two intertwined jobs
Prediction = evaluate a given policy (find \(v_\pi\)). Control = find the best policy (find \(v_*,\pi_*\)). Policy iteration and value iteration are two recipes for control built out of the prediction step.

2. Policy evaluation (prediction)

Repeatedly apply the Bellman expectation equation as an update until values stop changing:

\[ V_{k+1}(s)\leftarrow \sum_a \pi(a\mid s)\sum_{s',r}p(s',r\mid s,a)\big[r+\gamma V_k(s')\big] \]

For a deterministic policy this simplifies to one term per state: \(V(s)\leftarrow r + \gamma V(s')\) for the action \(\pi(s)\) dictates.

3. Policy improvement

Given \(v_\pi\), build a better policy by acting greedily on one-step look-ahead:

\[ \pi'(s)=\arg\max_a \sum_{s',r}p(s',r\mid s,a)\big[r+\gamma v_\pi(s')\big] \]
📈
Policy improvement theorem
Acting greedily w.r.t. \(v_\pi\) yields a policy \(\pi'\) that is no worse at every state: \(v_{\pi'}(s)\ge v_\pi(s)\). If it’s not strictly better anywhere, \(\pi\) was already optimal. This guarantee is what makes the next algorithm converge.

4. Policy iteration

Alternate the two steps until the policy stops changing:

1. Initialise π arbitrarily
2. EVALUATION : iterate V(s) ← Σ_a π(a|s) Σ p[r + γV(s')]  until V converges
3. IMPROVEMENT: π'(s) ← argmax_a Σ p[r + γV(s')]
4. if π' == π : stop (π is optimal)   else π ← π'  and go to 2
🔁
Generalised Policy Iteration (GPI)
Evaluation makes the value consistent with the policy; improvement makes the policy greedy w.r.t. the value. They pull toward each other and meet at the optimum \((v_*,\pi_*)\). Almost every RL method is an instance of this dance.

5. Value iteration

Don’t bother fully evaluating each policy — fold improvement straight into the update by taking the max:

\[ V_{k+1}(s)\leftarrow \max_a \sum_{s',r}p(s',r\mid s,a)\big[r+\gamma V_k(s')\big] \]

Iterate to convergence, then read off the greedy policy once. It is policy iteration with exactly one sweep of evaluation between improvements.

6. Synchronous vs. asynchronous (in-place)

SynchronousAsynchronous / in-place
Compute all new values from the old table, then overwrite together. Order doesn’t matter.Overwrite each state immediately; later states in the same sweep use the just-updated values. Order matters.
Dec 2025 Q1c says “synchronous”.Jul 2023 Q3 & Jul 2024-makeup Q2 say “in-place / asynchronous”, and give you the sweep order.
⚠️
Read the question’s update style first
In-place usually propagates information faster (a freshly-updated neighbour is used right away), but you must respect the stated order. Synchronous keeps a clean separation between iteration \(k\) and \(k{+}1\). Picking the wrong one is the most common arithmetic error on these papers.

8. When is a DP solution appropriate?

(Jan 2024 Q4c.) DP fits when:

  • The problem is a Markov decision process (current state summarises history).
  • A complete, accurate model \(p(s',r\mid s,a)\) is known.
  • State and action spaces are small enough to sweep exhaustively (no curse of dimensionality).

Examples: inventory control with known demand distribution; gambler’s problem; small gridworlds; the recycling robot. DP becomes infeasible for huge spaces (e.g. Go, raw-pixel games) — that is precisely where sampling methods (MC, TD) and function approximation (deep RL) take over.

📉
Scalability comment (Jul 2023 Q3c)
Both policy iteration and value iteration sweep every state each iteration, so cost grows with \(|\mathcal S|\times|\mathcal A|\). For large or continuous spaces this is intractable, and you need approximation / sampling instead. They also require the full model, which is often unavailable in practice.
